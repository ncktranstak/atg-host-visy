Sending TLS‑350 computer-format commands and raw serial data to a FAFNIR VISY‑Command automatic tank gauge from the ATG Host application

Abstract

ATG Host is a desktop application that communicates with a FAFNIR VISY‑Command automatic tank gauge over its RS‑232/RS‑485 host interface using the Veeder‑Root TLS‑350 computer format. Besides the built-in inventory and delivery polls, the application provides a manual command box that accepts any function code — and, using the hex: and text: prefixes, arbitrary raw data that is transmitted on the wire exactly as entered, without framing. This document specifies all three input forms, dissects the protocol frames with a fully decoded example exchange, lists the available commands, and gives ready-to-paste recipes for common diagnostic tasks.

The Command Panel

The left column of the main window contains the Commands group with three controls:

  • Inventory (i201) — sends an in-tank inventory request for the tank selected in the Polling group (All tanks → i20100, Tank 3 → i20103, …) and fills the Tanks tab from the reply.
  • Deliveries (i202) — the same for the delivery report; results appear in the Deliveries tab.
  • Manual command box — a free-form text box. Type an entry, press Enter or click Send. This box is the subject of this document.

Every transmission and every received frame is echoed in the Console tab with a millisecond timestamp, direction marker (TX/RX) and the frame content. Control characters are shown symbolically (<SOH>, <ETX>); ticking Hex view switches the console to a space-separated hexadecimal byte dump — the recommended view when working with raw data. Save log… writes the console content to a text file.

All entry forms work identically against real hardware and in Demo mode: the built-in simulator answers correctly framed requests it understands and replies with the standard rejection frame (§3) to anything else, so raw-data experiments can be rehearsed safely without a gauge attached.

Command Entry Forms

The manual box distinguishes three input forms by prefix. Whitespace around the entry is ignored; the prefixes are case-insensitive.

Function-code form (framed automatically)

An entry without a prefix is treated as a TLS‑350 function code. The application wraps it in the protocol's frame delimiters — SOH (0x01) in front, ETX (0x03) behind — and transmits it. Typing i20100 is therefore exactly equivalent to pressing the Inventory (i201) button with All tanks selected. This is the everyday form: you supply the six-character function code, the software supplies the framing.

Raw hexadecimal form — hex:

An entry starting with hex: is parsed as hexadecimal byte pairs and sent verbatim — no SOH, no ETX, no validation of any kind is added. Spaces and commas between pairs are ignored, so all of the following are the same eight bytes:

hex:01 69 32 30 31 30 30 03
hex:0169323031303003
hex:01,69,32,30,31,30,30,03
Equivalent hex: entries for the framed inventory request.

Because the frame delimiters are under your control in this form, you must include them yourself when you intend to send a valid request (note the leading 01 and trailing 03 above). Omitting them is equally legitimate — that is precisely how you test how the gauge treats line noise or fragmentary frames. An odd number of hex digits or a non-hex character is reported as !! invalid hex string and nothing is sent.

Raw text form — text:

An entry starting with text: sends the following characters as ASCII, again without any automatic framing. Two symbolic escapes are replaced before transmission, so the control characters can be typed on a keyboard:

Escapes recognised in text: mode.
EscapeByteMeaning
<SOH>0x01start of header (frame open)
<ETX>0x03end of text (frame close)

Thus text:<SOH>i20100<ETX> is byte-identical to the framed form and to the first hex: example, while text:i20100 sends the six naked characters with no delimiters — which a compliant gauge silently ignores or rejects. Non-ASCII characters are refused with a console message; use hex: for arbitrary byte values.

Note. Raw entries are deliberately passed through untouched. The application will happily transmit malformed, truncated or oversized data — that is the point of the feature. The console is your record of exactly what was sent; keep Hex view enabled while experimenting.

Frame Anatomy and a Decoded Exchange

A computer-format request is minimal: the function code between the two frame delimiters. There is no checksum on requests.

Request frame (8 bytes for a six-character function code).
BytesContent
01SOH
69 32 30 31 30 30i20100 — function i201, tank 00 (all)
03ETX

The response repeats the function code, adds a timestamp (YYMMDDHHmm), the data records, the delimiter && and a four-digit checksum: the 16-bit two's complement of the byte sum from SOH through && inclusive, so that (sum + checksum) mod 216 = 0. The application verifies it on every reply and prints !! checksum mismatch when it fails. An unknown or unsupported function is answered with the rejection frame <SOH>9999FF1B<ETX>.

The following exchange was captured from the application (demo mode, one tank); it is shown exactly as the console logs it:

TX  <SOH>i20100<ETX>
RX  <SOH>i2010026070216450110000007 464D767C 464DAB1A 4683A4C2
    4488F9A8 3E94597A 416172F0 405E8637 &&EE42<ETX>

Data fields are IEEE-754 single-precision floats transmitted as eight hexadecimal digits of the big-endian bit pattern. The Tanks and Deliveries tabs perform this decoding automatically for i201/i202; for any other function the raw payload remains visible in the console.

Command List

A function code has the form iFFFTT: three digits selecting the report, two digits addressing the tank. Tank 00 addresses all tanks; 01 to 16 address one tank.

Further TLS-350 computer-format functions commonly implemented by compatible gauges. Support varies with the gauge model and the host code configured with VISY-Setup — probe them from the manual box; an unsupported function is answered with 9999FF1B and does no harm.

Responses to functions other than i201/i202 are not decoded into tables, but the full frame — timestamp, payload and checksum verdict — is always available in the console, which makes the manual box a serviceable protocol explorer.

Warning. Raw mode can transmit anything, including byte sequences a gauge firmware may misinterpret. On a live station, restrict experiments to read-only report functions (i…); never improvise write or configuration commands against production equipment.

Troubleshooting

Console symptoms and their causes.
SymptomCause and remedy
RX <SOH>9999FF1B<ETX>The gauge rejected the function: unknown code, unsupported report, or the entry was misspelt. Check the function code and the host protocol configured with VISY-Setup.
!! checksum mismatchThe reply arrived corrupted — wrong baud rate/parity, cable interference, or a shared bus collision. Verify the line settings match the gauge (default 9600 8N1).
no reply at allFraming missing in a raw entry (SOH/ETX), wrong COM port, TX/RX swapped, or the host interface disabled on the gauge. The 1 s stale-buffer flush will print any partial bytes that did arrive.
!! invalid hex stringThe hex: entry contains an odd digit count or non-hex characters; nothing was sent.
!! text mode accepts ASCII onlyThe text: entry contains characters above 0x7F; use hex: instead.
garbled console textBinary reply bytes shown in character view — enable Hex view.
